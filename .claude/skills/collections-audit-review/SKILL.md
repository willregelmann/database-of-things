# Collections Audit Review

Second-pass review of the collections-audit-fix job's own output: the PRs
`submit.mjs` opens from a run's changelog, and the issues it files from
`flag_finding` calls, both labeled `audit-finding`. Where `collections-curate`
and `collections-audit-fix` must always stop short of `git push`/`gh pr
create`/merging, **this skill is the one standing exception** — but the
exception is scoped exactly to `audit-finding` PRs/issues that were
themselves machine-generated by that job, never anything else.

Two independent flows:

- **Flow A — review a PR**: re-verify every changed field cited in the PR
  body against a fresh source; merge if it all holds up and CI is green,
  otherwise comment and leave it for a human.
- **Flow B — resolve an issue**: re-verify the flagged finding; if it holds
  up *and* the fix only requires applying an already-documented convention
  (not inventing a new one), write the fix directly (this skill has
  Read/Write/Edit/Bash, unlike the MCP-only tool surface `collections-audit-fix`
  is restricted to), open a PR, and merge it. Otherwise comment and leave the
  issue open.

## Why re-verification, not a rubber stamp

The PR/issue already cites a source — that source came from the same
`collections-audit-fix` session that made the claim, so trusting it here
would just be trusting the thing being reviewed. Every claim gets checked
against a **fresh, independently-run search**, not "does the cited URL say
what the PR claims it says." A misread page or a hallucinated source is
exactly the failure mode a second pass exists to catch.

## Hard rules

- **Scope lock.** Only ever act on a PR whose head branch matches
  `^audit/[0-9a-f]{8}$` *and* carries the `audit-finding` label, or an issue
  that carries the `audit-finding` label *and* whose body contains the
  literal marker `Filed automatically by the collections-audit-fix job`
  (see `buildIssue` in `tools/collections-mcp/submit.mjs`). A label alone
  isn't enough — both signals together confirm this is genuinely
  machine-generated audit-fix output, not something a human separately
  labeled. Never touch any other PR, branch, or issue, for any reason.
- **The git/gh authority this skill has is scoped, not general.** Merging,
  pushing, and opening PRs here is allowed *only* for the scope-locked items
  above. It is not a general grant of git autonomy to this repo — every
  other skill's "ask first" rule still applies to everything else.
- **Never edit an already-opened audit-fix PR's branch.** Flow A is
  verify-then-merge-or-comment, never verify-then-fix-and-merge — if part of
  a PR is wrong, that PR doesn't merge at all; say exactly which line(s)
  failed and why, in a comment, and leave it for a human. Partial-correctness
  edits blur who's responsible for what shipped.
- **A fix for an issue may only apply an already-documented convention** —
  naming/slugification rules, the tag-creation process in `tags/CLAUDE.md`,
  an existing category's components-bucket pattern, a nested collection's
  directory-position-is-parentage rule, etc. If resolving a finding would
  mean inventing a new convention (a first-of-its-kind category's
  `CLAUDE.md`/`template.schema.json`, a novel taxonomy call with no
  documented precedent, a new tag namespace beyond `franchises/`): that's
  `collections-curate` territory requiring real curatorial judgment, not
  this skill's — comment with what's needed and why, leave the issue open.
- **Cite a source for every verification, and leave alone what you can't
  confirm.** Same discipline as `collections-audit-fix`: an unconfirmed
  guess is worse than leaving the gap. "Inconclusive" is a real outcome,
  distinct from "confirmed wrong" — see Step 2/3.
- **CI must be green before any merge.** Check
  `gh pr view <n> --json statusCheckRollup` (or, for a PR you just opened
  in Flow B, wait for it to complete). Never `--admin`, never merge past a
  failing or pending check.
- **Never force-push, never touch branch protection, never close an issue
  without either merging a fix for it or explaining why it's being closed
  unfixed** (Step 3's "confirmed wrong" outcome).
- **Leave a marker comment on everything you touch, including a clean
  merge**, and **skip anything that already has one** (Step 0). This is
  the only thing preventing every scheduled run from redoing the same web
  searches on the same backlog forever.
- **Renames get extra scrutiny.** `collections-audit-fix`'s own history
  includes a rename that looked right by surface pattern-matching but
  broke correct data because the two files were never actually the same
  card the way a sibling pair was — verify a rename's *target*, not just
  that a rename happened, the same way you'd verify any other fact.
- **Large PRs get a disclosed sample, not silence.** See Step 1's sampling
  rule — never silently skip verifying most of a big PR without saying so
  in the merge comment.
- **Git hygiene**: `git fetch origin main` and `git checkout --detach
  origin/main` before starting *each* item (PR or issue), not just once at
  the top of the run — a merge from a prior item in the same run moves
  `main`, and a stale local `tools/collections-validate/validate.mjs` can
  pass something a fresh one (what CI actually runs) would reject. This
  mirrors why `run-audit-job.sh` re-syncs the audit-fix job's checkout the
  same way.

## Step 0: Gather and filter the queue

```bash
gh pr list --label audit-finding --state open --json number,headRefName,title,body,comments
gh issue list --label audit-finding --state open --json number,title,body,comments
```

For each result:

1. Apply the scope-lock filter (Hard rules) — drop anything that doesn't
   match.
2. Drop anything whose `comments` already contains a comment starting with
   `_Reviewed by collections-audit-review` (the marker this skill's own
   comments always start with — see Step 4). Already reviewed once; a
   human owns it now, not a rerun of this skill.

What's left is this run's actual worklist. If it's empty, say so and stop —
don't manufacture something to do.

## Step 1: Flow A — review a PR

For each PR left in the worklist:

1. `git fetch origin main && git checkout --detach origin/main`, then
   inspect the PR's actual diff (`gh pr diff <n>`) — the source of truth for
   what changed, not just the mechanically-generated body text (though the
   body's per-field `from → to` listing is a convenient index into it).
2. **Verify every entry, unless the PR is large.** "Large" means more than
   15 changed entries (added/updated/renamed). For a PR at or under that
   size, verify every entry. For a larger one, verify a random sample of at
   least 15 entries or 20% of the total (whichever is bigger), always
   including every rename entry (extra scrutiny — see Hard rules) — and say
   in the merge/comment (Step 4) exactly what was sampled, e.g. "verified
   18 of 95 changed entries (all renames + a random sample), all held up."
3. For each entry being verified: fresh web search for the item/collection
   it names, and confirm the **new** value is what an authoritative source
   actually says. An "added" entry needs the whole record confirmed to
   exist and be accurate, not just one field.
4. Check CI: `gh pr view <n> --json statusCheckRollup` — every check must
   be `SUCCESS`/`COMPLETED`. If anything is pending, it's fine to wait a
   short while in the same pass; if still not green, treat as failing.
5. Decide:
   - **All sampled/verified entries hold up, CI green** → merge:
     `gh pr merge <n> --merge` (repo default is a merge commit, matching
     every prior merged audit PR — not squash, not rebase). Leave the Step
     4 marker comment first or as part of the merge; either order is fine
     as long as it lands.
   - **Any entry doesn't hold up** → do not merge. Leave the Step 4 marker
     comment identifying exactly which entry/entries failed, what the
     independent search actually found, and why. Leave the PR open for a
     human.
   - **CI failing/red** → do not merge. Marker comment noting the failing
     check; leave for a human.

## Step 2: Flow B — resolve an issue

For each issue left in the worklist:

1. `git fetch origin main && git checkout --detach origin/main`.
2. Read the issue body: the specific finding, its cited source(s), the
   collection it's about, and why `collections-audit-fix`'s tool surface
   couldn't cover it.
3. Fresh web search to independently verify the finding. Three possible
   outcomes:
   - **Confirmed true.** Continue to step 4.
   - **Confirmed false** (independent search contradicts the finding, not
     just "couldn't find confirmation") — close the issue with a marker
     comment explaining what the independent search found and why the
     original finding doesn't hold. This is different from "inconclusive"
     below — only use it when you're actually confident it's wrong, same
     bar as confirming something's right.
   - **Inconclusive** (can't authoritatively confirm either way) — marker
     comment noting what was searched and that it's inconclusive; leave
     the issue open, don't act further.
4. Check whether resolving it only requires an already-documented
   convention (Hard rules). If it would require inventing a new one —
   marker comment explaining what's missing and why it's out of this
   skill's scope; leave the issue open.
5. Otherwise, make the fix directly, following the same conventions
   `collections-curate` documents (naming, `id` generation via `uuidgen`,
   schema conformance, tag-creation process, etc.):
   - New/moved/renamed entity files, new tag entities under `tags/`, a new
     components bucket following an existing category's pattern, etc. —
     whatever the specific finding needs, using Read/Write/Edit directly
     (this skill isn't restricted to the `collections-mcp` tool surface).
   - Run `cd tools/collections-validate && npm run validate` — fix any
     errors before continuing.
6. Branch, commit, push, open the PR:
   ```bash
   git checkout -b "audit-review/$(uuidgen | cut -c1-8)"
   git add <paths>
   git commit -m "<title>

   Fixes #<issue-number>

   <body: what changed, citing the fresh verification source(s)>"
   git push -u origin <branch>
   gh pr create --title "<title>" --body "<body, same as commit>" --head <branch>
   ```
   `Fixes #<n>` auto-closes the issue when the PR merges — that's the
   intended way this issue closes, not a separate manual close.
7. Wait for CI, same bar as Flow A step 4. If green, merge the same way
   (`gh pr merge <n> --merge`) and leave the Step 4 marker comment on the
   now-merged PR. If CI fails, leave the marker comment on the PR (not the
   already-linked issue) explaining what failed, and leave both open for a
   human — don't force a merge to make the `Fixes #n` auto-close happen.
8. `git checkout --detach origin/main` and delete the local branch before
   moving to the next item in the worklist.

## Step 3: Report

Summarize the run: how many PRs/issues were in scope, how many were
skipped (already reviewed), and for each one actually processed — what
happened (merged / commented-and-left / closed-as-unconfirmed) and why in
one line. Don't just say "reviewed N items" — a human scanning this report
should be able to tell which ones need their attention without reopening
every PR/issue themselves.

## Step 4: The marker comment

Every PR or issue this skill touches gets exactly one comment, starting
with the literal text `_Reviewed by collections-audit-review —` followed
by the verdict and a one-line reason (which entries/what was sampled for a
PR; what the fresh search found for an issue). This is both the audit
trail and the dedup key Step 0 uses to skip already-reviewed items on the
next run — never touch something that already has one, and never merge or
close without leaving one.

## Checklist

- [ ] Applied the scope lock (branch pattern + label, or label + issue-body
      marker) before touching anything
- [ ] Skipped anything already carrying a `_Reviewed by
      collections-audit-review` comment
- [ ] Re-verified with a *fresh* search — never treated the PR/issue's own
      cited source as sufficient on its own
- [ ] Applied the large-PR sampling rule and disclosed the sample in the
      comment, rather than silently checking less than claimed
- [ ] Gave every rename extra scrutiny
- [ ] Never edited an already-opened audit-fix PR's branch
- [ ] Every issue-fix only applied an already-documented convention — no
      new `CLAUDE.md`/`template.schema.json`/taxonomy call invented here
- [ ] Confirmed CI green before every merge
- [ ] Re-synced to `origin/main` before starting each item, not just once
- [ ] Left exactly one marker comment on everything touched
